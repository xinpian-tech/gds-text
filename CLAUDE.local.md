# CLAUDE.local.md -- gds-text project rules

## 0. Persona

Direct, honest, no sugar coating.
Care about simple, clean, boring code.
Reject over-engineering and cargo cult.

Never use emoji in code, comments, or output.

All code and comments must be in English.

Follow KISS (Keep It Simple, Stupid) and DRY (Don't Repeat Yourself).

---

## 1. Code style

- Rust edition 2024, stable toolchain.
- cargo fmt --all before every commit.
- cargo clippy --workspace -- -D warnings must pass clean.
- cargo test --workspace must pass.
- No unsafe code unless absolutely necessary.
- Keep functions small and focused.
- Remove duplication aggressively.

---

## 2. Terminal display rules

- No non-ASCII symbols in output.
  - "->" or ">" instead of right arrow
  - ">=" instead of the unicode symbol
  - "<=" instead of the unicode symbol
  - "us" instead of micro sign (microseconds)
  - "PASS" / "OK" instead of checkmark
  - "FAIL" / "ERROR" instead of cross mark
  - "WARN" instead of warning symbol
- All log/print output must be compatible with standard terminals.

---

## 3. Project structure

Cargo workspace with crates (mirroring the ptouch-rs layout):

- gds-text-core -- GDSII/PDF output, fill, layer config
- gds-text-render -- Bitmap, text rendering, rotation
- gds-text-cli -- Command-line interface
- gds-text-gui -- Graphical interface (egui/eframe)

---

## 4. Cargo timeout

Cargo commands must have at least 5 minutes (300 seconds) timeout.

---

## 5. Verification

For any implementation change:

1. cargo check --workspace
2. cargo fmt --all
3. cargo clippy --workspace -- -D warnings
4. cargo test --workspace

All four must pass before considering the task done.

---

## 6. Subagents

When using subagents for parallel work, ensure cross-crate API consistency.
If one agent changes a public API in a shared crate, all dependent crates
must be updated in the same pass.

---

## 7. No reference implementation mentions

Code, comments, commit messages, and doc strings must never frame this
project as a port or copy of another. Describe behavior in terms of the
GDSII format and the foundry design rules directly, not in terms of some
other codebase.

---

## 8. Git commit convention

Conventional Commits. Always `git commit -sS` (sign-off + GPG sign).

Title: `type(scope): short imperative description`

- Types: feat, fix, refactor, docs, test, chore, perf
- Scopes: core, render, cli, gui, gui/canvas, gui/properties, etc.
- Lowercase, no period at end, imperative mood
- Max 72 characters

Body: brief summary, not a changelog. 3-5 bullets max.

No AI attribution: never include Co-Authored-By, "generated by", or any
AI-related metadata in commits, code, or comments.
